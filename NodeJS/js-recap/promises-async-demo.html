What are Promises?

A Promise in JavaScript represents the eventual completion (or failure) of an asynchronous operation and its resulting value. Think of it as a placeholder for a value that will be available in the future.

üçï Pizza Delivery Analogy

Imagine ordering a pizza online:

  • Promise Created: You place the order and get a receipt (promise)
  • Pending: The pizza is being prepared
  • Fulfilled: Pizza arrives successfully
  • Rejected: Restaurant calls saying they're out of ingredients

A Promise has three states:

  • ‚è≥ Pending - Initial state, neither fulfilled nor rejected
  • ‚úÖ Fulfilled - Operation completed successfully
  • ‚ùå Rejected - Operation failed

Creating and Using Promises

üîß Basic Promise Example

// Creating a simple Promise
const fetchUserData = new Promise((resolve, reject) => {
    // Simulate API call with setTimeout
    setTimeout(() => {
        const success = Math.random() > 0.3; // 70% success rate
        
        if (success) {
            resolve({ name: "John Doe", id: 123 });
        } else {
            reject(new Error("Failed to fetch user data"));
        }
    }, 2000);
});

// Using the Promise with .then() and .catch()
fetchUserData
    .then(userData => {
        console.log("Success:", userData);
    })
    .catch(error => {
        console.log("Error:", error.message);
    });

Promise Chaining

Promises can be chained together using .then() to perform sequential operations.

üè≠ Assembly Line Analogy

Think of promise chaining like an assembly line where each worker (promise) completes their task and passes the result to the next worker.

üîó Promise Chaining Example

// Chaining promises for sequential operations
function getUser(id) {
    return new Promise(resolve => {
        setTimeout(() => resolve({ id, name: "User " + id }), 1000);
    });
}

function getUserPosts(user) {
    return new Promise(resolve => {
        setTimeout(() => resolve([
            "Post 1 by " + user.name,
            "Post 2 by " + user.name
        ]), 1000);
    });
}

function getPostComments(posts) {
    return new Promise(resolve => {
        setTimeout(() => resolve({
            posts,
            comments: ["Great post!", "Thanks for sharing!"]
        }), 1000);
    });
}

// Chaining the promises
getUser(1)
    .then(user => getUserPosts(user))
    .then(posts => getPostComments(posts))
    .then(result => console.log("Final result:", result))
    .catch(error => console.log("Error:", error));

Async/Await - The Modern Approach

async/await is syntactic sugar built on top of Promises. It makes asynchronous code look and behave more like synchronous code, making it easier to read and write.

üìñ Reading a Book Analogy

Promises with .then() are like reading a book where each chapter tells you "turn to page X for the next part." Async/await is like reading a normal book where you just continue to the next page naturally.

üîÑ With Promises (.then)

fetchData()
  .then(data => processData(data))
  .then(processed => saveData(processed))
  .then(result => console.log(result))
  .catch(error => console.log(error));

‚ú® With Async/Await

async function handleData() {
  try {
    const data = await fetchData();
    const processed = await processData(data);
    const result = await saveData(processed);
    console.log(result);
  } catch (error) {
    console.log(error);
  }
}

üöÄ Async/Await Example

// Same functionality as promise chaining, but cleaner syntax
async function fetchUserDataAsync() {
    try {
        console.log("üîç Fetching user...");
        const user = await getUser(1);
        
        console.log("üìù Fetching posts...");
        const posts = await getUserPosts(user);
        
        console.log("üí¨ Fetching comments...");
        const result = await getPostComments(posts);
        
        console.log("‚úÖ All data fetched successfully!");
        return result;
        
    } catch (error) {
        console.log("‚ùå Error occurred:", error.message);
        throw error;
    }
}

// Call the async function
fetchUserDataAsync();

Error Handling Comparison

üö® Error Handling Demo

Let's see how both approaches handle errors:

Promise Error Handling

// Using .catch()
function promiseErrorDemo() {
    return new Promise((resolve, reject) => {
        setTimeout(() => {
            reject(new Error("Something went wrong!"));
        }, 1000);
    })
    .catch(error => {
        console.log("Caught error:", error.message);
        return "Default value";
    });
}

Async/Await Error Handling

// Using try/catch
async function asyncErrorDemo() {
    try {
        await new Promise((resolve, reject) => {
            setTimeout(() => {
                reject(new Error("Something went wrong!"));
            }, 1000);
        });
    } catch (error) {
        console.log("Caught error:", error.message);
        return "Default value";
    }
}

Parallel vs Sequential Execution

Understanding when to run operations in parallel vs sequentially is crucial for performance.

üèÉ‚Äç‚ôÇÔ∏è Sequential vs Parallel Demo

// Sequential execution (slower)
async function sequentialDemo() {
    const start = Date.now();
    
    const result1 = await simulateAPICall("API 1", 2000);
    const result2 = await simulateAPICall("API 2", 2000);
    const result3 = await simulateAPICall("API 3", 2000);
    
    const duration = Date.now() - start;
    return { results: [result1, result2, result3], duration };
}

// Parallel execution (faster)
async function parallelDemo() {
    const start = Date.now();
    
    const [result1, result2, result3] = await Promise.all([
        simulateAPICall("API 1", 2000),
        simulateAPICall("API 2", 2000),
        simulateAPICall("API 3", 2000)
    ]);
    
    const duration = Date.now() - start;
    return { results: [result1, result2, result3], duration };
}

üéØ Key Takeaways

‚ú® When to Use Async/Await

  • When you want cleaner, more readable code
  • For complex sequential operations
  • When you need better error handling with try/catch
  • For debugging (easier to set breakpoints)

üîó When to Use Promises

  • When working with older codebases
  • For simple one-off async operations
  • When you need Promise.all, Promise.race, etc.
  • In callback-heavy environments
