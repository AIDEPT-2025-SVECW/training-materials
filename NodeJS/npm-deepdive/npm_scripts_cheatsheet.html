üìö What are NPM Scripts?

NPM scripts are commands defined in your package.json file that automate repetitive tasks in your development workflow. They're like shortcuts or macros for common operations.

üéØ Analogy: Think of NPM scripts as speed dial on your phone. Instead of manually typing a long phone number every time, you assign it to a button and just press that button. Similarly, instead of typing long terminal commands repeatedly, you create a short script name that runs those commands for you!
Why Use NPM Scripts?
  • Standardize commands across your team
  • Save time typing repetitive commands
  • Document common tasks in your project
  • Chain multiple commands together
  • Create a consistent development workflow

üìù Basic package.json Structure

Here's how scripts are defined in your package.json:

{
  "name": "my-awesome-project",
  "version": "1.0.0",
  "scripts": {
    // Simple script - just runs a command
    "start": "node app.js",
    
    // Development script with watch mode
    "dev": "nodemon app.js",
    
    // Run tests
    "test": "jest",
    
    // Build for production
    "build": "webpack --mode production"
  }
}

Running Scripts:

# Standard way for any script
npm run dev

# Some scripts have shortcuts (start, test, stop, restart)
npm start      # Same as: npm run start
npm test       # Same as: npm run test

üîó Chaining Commands

You can chain multiple commands together to create powerful automation workflows:

Sequential (&&)

Run commands one after another. If one fails, the rest don't run.

"build": "npm run clean && npm run compile"

Parallel (&)

Run multiple commands at the same time.

"dev": "npm run watch-css & npm run watch-js"

Pipe (|)

Pass output from one command to another.

"analyze": "webpack --json | webpack-bundle-analyzer"
üéØ Analogy: Think of command chaining like a recipe:
  • Sequential (&&): "Chop vegetables AND THEN cook them" - order matters!
  • Parallel (&): "Boil water WHILE chopping vegetables" - multitasking!
  • Pipe (|): "Strain pasta THEN pour sauce on it" - output flows to next step

‚ö° Advanced Script Examples

{
  "scripts": {
    // Clean build directory before building
    "clean": "rm -rf dist",
    "prebuild": "npm run clean",
    "build": "webpack --mode production",
    "postbuild": "echo 'Build completed!'",
    
    // Development with automatic restart
    "dev": "nodemon --watch src --exec 'node app.js'",
    
    // Run tests with coverage
    "test": "jest",
    "test:watch": "jest --watch",
    "test:coverage": "jest --coverage",
    
    // Linting and formatting
    "lint": "eslint src/**/*.js",
    "lint:fix": "eslint src/**/*.js --fix",
    "format": "prettier --write 'src/**/*.js'",
    
    // Combined workflow
    "check": "npm run lint && npm run test",
    "deploy": "npm run check && npm run build && git push heroku main"
  }
}
üé™ Lifecycle Hooks: NPM automatically runs scripts with special names:
  • pre{script}: Runs BEFORE the main script
  • post{script}: Runs AFTER the main script
  • Example: Running npm run build automatically runs: prebuild ‚Üí build ‚Üí postbuild

üõ†Ô∏è Practical Use Cases

Development Server

"dev": "nodemon --watch src app.js"

Automatically restarts server when files change

Code Quality

"precommit": "npm run lint && npm test"

Ensure code quality before commits

Multiple Environments

"start:prod": "NODE_ENV=production node app.js"
"start:dev": "NODE_ENV=development nodemon app.js"

Different configs for different environments

Database Tasks

"db:seed": "node scripts/seed.js"
"db:migrate": "node scripts/migrate.js"

Automate database operations

üí° Best Practices

  • Use descriptive names: test:unit is better than tu
  • Document complex scripts: Add comments explaining what they do
  • Keep scripts simple: For complex logic, create a separate script file and call it
  • Use cross-platform commands: Or use packages like cross-env for compatibility
  • Group related scripts: Use prefixes like test:*, build:*
