üéØ What is Semantic Versioning (SemVer)?

Semantic Versioning is a versioning scheme that uses a three-part number: MAJOR.MINOR.PATCH (e.g., 2.4.1)

üè† House Analogy: Think of software versions like renovating a house:
  • MAJOR (2.x.x): Tearing down walls and rebuilding - breaks compatibility
  • MINOR (x.4.x): Adding a new room - new features, still compatible
  • PATCH (x.x.1): Fixing a leaky faucet - bug fixes only

üìã The Three Rules of SemVer

1. MAJOR Version (Breaking Changes)

Increment when you make incompatible API changes that break existing functionality.

// Version 1.x.x - Old API
function calculateTotal(price) {
    return price * 1.1; // Returns number
}

// Version 2.0.0 - Breaking change!
function calculateTotal(items) {
    return {  // Now returns object
        subtotal: items.reduce((a, b) => a + b, 0),
        tax: 0.1
    };
}

2. MINOR Version (New Features)

Increment when you add functionality in a backward-compatible manner.

// Version 1.0.0 - Original
class User {
    constructor(name) {
        this.name = name;
    }
}

// Version 1.1.0 - Added new method (backward-compatible)
class User {
    constructor(name) {
        this.name = name;
    }
    
    getGreeting() {  // New feature!
        return `Hello, ${this.name}!`;
    }
}

3. PATCH Version (Bug Fixes)

Increment when you make backward-compatible bug fixes.

// Version 1.1.0 - Bug present
function formatDate(date) {
    return date.getMonth() + '/' + date.getDate();  // Missing year!
}

// Version 1.1.1 - Bug fixed
function formatDate(date) {
    return date.getMonth() + '/' + date.getDate() + '/' + date.getFullYear();
}

  "name": "my-awesome-app",
  "version": "1.0.0",
  "dependencies": {
    "express": "^4.18.0",     // Accept 4.18.0 to <5.0.0
    "lodash": "~4.17.21",    // Accept 4.17.21 to <4.18.0
    "react": "18.2.0",       // Exact version only
    "axios": "*"            // Latest version (not recommended!)
  }
}

‚ö° NPM Version Commands

Automatic Version Bumping:

# Increment patch version (1.0.0 ‚Üí 1.0.1)
npm version patch

# Increment minor version (1.0.1 ‚Üí 1.1.0)
npm version minor

# Increment major version (1.1.0 ‚Üí 2.0.0)
npm version major

# Set specific version
npm version 2.5.1

# Add pre-release identifier
npm version prerelease --preid=beta  # ‚Üí 1.0.0-beta.0

Installing Specific Versions:

# Install latest version
npm install express

# Install specific version
npm install express@4.18.0

# Install with version range
npm install express@^4.18.0

# Update packages within allowed ranges
npm update

# Check for outdated packages
npm outdated

‚úÖ Best Practices

üìö Remember:
  • Start at 1.0.0 for your first stable release
  • Use 0.x.x for initial development (anything can change)
  • Never change published versions - always increment
  • Document breaking changes in CHANGELOG.md
  • Use ^ for dependencies in most cases (safe updates)
  • Lock versions with package-lock.json for consistency
