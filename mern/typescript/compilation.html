Core Concepts

Compilation & Type Inference

Understand how TypeScript turns into JavaScript, and how it's smart enough to know your types even without you typing them.

⚙️ 1. Compilation: From TS to JS

Browsers only understand standard JavaScript. The TypeScript compiler (tsc) is responsible for taking your .ts code, checking it for errors, and converting it to .js.

Compiling a single file

If you have a file named app.ts, you can compile it manually from the terminal:

Terminal
tsc app.ts

This generates an app.js file next to it. But doing this manually for every file is tedious.

Watch Mode

You can tell the compiler to "watch" your file for changes and recompile automatically whenever you hit Save.

Terminal
tsc app.ts -w   // or --watch

The tsconfig.json File

When you are working on a real project (not just a single file), you don't compile files one by one. You use a configuration file called tsconfig.json.

You generate this file once per project:

Terminal
tsc --init

Once you have this file, you can simply run tsc (without any file names) and it will compile your entire project based on the rules in the config!

Crucial Fact: By default, TypeScript will still generate the .js file even if your .ts code has type errors! It assumes you might be in the middle of migrating. You can stop this by setting "noEmitOnError": true in your tsconfig.json.

🧪 Mini-Experiment: See it for yourself

Let's test both the positive (success) and negative (failure) scenarios of compilation!

Step 1 (Positive): Create a file named test.ts and add valid code.

let message: string = "All good!";
console.log(message);

Run tsc test.ts in your terminal. Notice how it quietly succeeds and creates test.js.

Step 2 (Negative): Now, let's intentionally break the code.

let age: number = "Twenty"; // Error! String assigned to number.

Run tsc test.ts again. You will see a red error in your terminal: error TS2322: Type 'string' is not assignable to type 'number'. However, check your folder — test.js was still generated because of the crucial fact mentioned above!

🧠 2. Type Inference

A common misconception about TypeScript is that you have to write explicit types for everything. This is false. TypeScript has a powerful engine called Type Inference.

Type inference means TypeScript will guess the type of a variable based on the value you assign to it.

Implicit vs Explicit Typing

app.tsTypeScript
// ❌ Explicit Typing (Redundant)
let username: string = "Alice";

// ✅ Implicit Typing (Type Inference)
let score = 100; // TS automatically knows this is a 'number'

// It will still catch errors!
score = "High"; // Error: Type 'string' is not assignable to type 'number'.

Because you assigned 100 to score when you created it, TypeScript locked its type as number. You don't need to type : number.

When should you write Explicit Types?

If TypeScript is so smart, when do we actually need to write types ourselves?

Scenario What to do?
Variable initialization (let x = 10;) Let TypeScript infer it.
Function Parameters Always explicit. TS cannot guess what you intend to pass into a function.
Function Return Types Optional, but highly recommended. Explicit return types prevent you from accidentally returning the wrong thing.
Delayed Initialization Explicit. If you declare a variable but don't assign it immediately, TS infers any (which is bad).
app.tsTypeScript
// Example of Delayed Initialization
let userStatus; // Type inferred as 'any'
userStatus = "Online";
userStatus = 1; // Valid, because it is 'any'

// How to fix it:
let correctStatus: string; // Explicit type
correctStatus = "Online";
correctStatus = 1; // Error! Catches the bug.
