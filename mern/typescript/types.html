TypeScript Foundations

Primitives & Special Types

The building blocks of data. Let's explore how TypeScript models everyday information.

The Big Three (Primitives)

string

Textual Data

Used for any text, surrounded by single quotes, double quotes, or backticks (template literals).

🌎 Real World Use Cases: Usernames, Email addresses, API endpoints, User IDs, HTML template strings.
types.ts
let apiEndpoint: string = "https://api.github.com";
let greeting: string = `Fetching from ${apiEndpoint}`;
number

Numeric Data

In TypeScript (and JS), there are no separate types for integers or floats. Everything is a number.

🌎 Real World Use Cases: Product prices, Coordinates (Lat/Lng), Age, Database row counts, Timestamps.
types.ts
let price: number = 99.99;
let totalViews: number = 1500000;
let hexColor: number = 0xffffff;
boolean

Logical Data

Represents a simple true or false value. The foundation of all logic and conditional rendering.

🌎 Real World Use Cases: isLoggedIn, isLoading, hasError, Feature toggles.
types.ts
let isAuthenticated: boolean = true;
let hasPaid: boolean = false;
Experiment #1

The Primitive Collision

What happens if you try to combine these types incorrectly? Let's write an experiment.

Create experiment.ts and add this code:

experiment.ts
let accountBalance: number = 1000;

// Imagine this comes from an HTML input field (which always returns strings)
let depositAmount: string = "500";

// Try to assign the string to the number variable
accountBalance = depositAmount;

🛠 Run tsc experiment.ts

error TS2322: Type 'string' is not assignable to type 'number'.

Takeaway: TypeScript prevents classic JS bugs (where "1000" + "500" = "1000500"). It forces you to explicitly convert the string to a number using Number(depositAmount) or parseInt().

The Special Types

Beyond primitives, TypeScript provides special types to handle unique situations like dynamic data, absent data, or impossible states.

any

The Escape Hatch

Turns off TypeScript's checking entirely for a variable. It can be anything.

🌎 Use sparingly when: Migrating old JS code rapidly, or when dealing with a highly dynamic external library that has no types.
types.ts
let mystery: any = "hello";
mystery = 123;     // ✅ Allowed
mystery.foo();     // ✅ Allowed (will crash at runtime though!)
unknown

The Safe 'any'

Similar to any, but safe. You must prove what type it is before you can use it.

🌎 Real World Use Cases: Parsing JSON.parse() payloads, fetching data from unknown 3rd party APIs.
types.ts
let data: unknown = fetchSomeData();

data.toUpperCase(); // ❌ Error! TS won't let you.

if (typeof data === "string") {
    data.toUpperCase(); // ✅ OK! We proved it's a string.
}
void / never

Nothingness

void: A function that returns nothing.
never: A function that never finishes (e.g. infinite loop or always throws an error).

🌎 Real World Use Cases: void for Event listeners (onClick). never for centralized error handling functions.
types.ts
function logData(msg: string): void {
    console.log(msg);
}

function crash(): never {
    throw new Error("Boom!");
}
Experiment #2

The 'any' vs 'unknown' Test

Let's prove why unknown is safer than any.

experiment.ts
// Scenario A: The reckless 'any'
let userInputA: any = 500;
// Try to use a string method on a number. TS says nothing! 
// This will CRASH when you run node!
let lenA = userInputA.length; 

// Scenario B: The strict 'unknown'
let userInputB: unknown = 500;
// Try to use a string method...
let lenB = userInputB.length;

🛠 Run tsc experiment.ts

error TS2571: Object is of type 'unknown'.

Takeaway: any completely defeats the purpose of TypeScript. unknown forces you to write defensive code (like checking typeof) before interacting with dynamic data.
