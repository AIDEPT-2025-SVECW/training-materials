📘 TypeScript Modules & Namespaces

Learn the concepts visually with practical examples, comparisons, and modern TypeScript practices.

Definitions & Core Motivation

Namespaces (Formerly Internal Modules)

A TypeScript-specific way to organize code within a single global scope or across multiple files, preventing name collisions by grouping related code into named JavaScript objects.

Modules (External Modules)

The standard ES6 implementation where each file has its own isolated scope. Anything declared inside a module remains private unless explicitly exported.

💡 The Real-World Analogy

Namespace: Think of a multi-shelf tool toolbox. All tools are in one physical box, but labeled trays keep the metric sockets separate from imperial sockets so they don't mix up.

Module: Think of separate shipping containers. Each container is sealed completely. To use a tool from another container, it must be explicitly shipped (exported) and accepted (imported) through defined customs checkpoints.

Why Do We Need Them?

  • Avoid Global Scope Pollution: Prevents overwriting functions or types with identical names across large teams.
  • Maintainability & Encapsulation: Groups domain logic into modular units and exposes only public interfaces.
  • Dependency Management: Modules allow build engines (like Webpack, Turbopack, or Vite) to construct explicit dependency graphs and perform tree-shaking.

Namespaces in Action

Namespaces execute at runtime as standard JavaScript IIFEs (Immediately Invoked Function Expressions). They are useful for grouping utility logic or legacy code, but are rarely used in modern React/Next.js codebases.

// Define a namespace for Payment Processing
namespace PaymentGateway {
  // Private internal type (Not accessible outside)
  type APIKey = string;
  const secretKey: APIKey = "sk_test_998877";

  // Public interface (Accessible outside via export)
  export interface TransactionResponse {
    success: boolean;
    transactionId: string;
  }

  // Public utility function
  export function processPayment(amount: number): TransactionResponse {
    console.log(`Processing $${amount} using key ${secretKey}`);
    return { success: true, transactionId: "tx_12345" };
  }
}

// Usage: Access via dot-notation
const result = PaymentGateway.processPayment(150);
// PaymentGateway.secretKey; // ❌ Error: 'secretKey' is not exported.

ES Modules in Action

Modern TypeScript development defaults to ES Modules. Every file containing an import or export keyword is processed as an isolated module.

File 1: auth.ts (Module)

// Local private state
const AUTH_ENDPOINT = "/api/v1/auth";

export interface User {
  id: string;
  name: string;
}

export function login(username: string): User {
  return { id: "u_1", name: username };
}

File 2: app.ts (Module Consumer)

import { login, User } from "./auth";

const currentUser: User = login("alex_dev");
// console.log(AUTH_ENDPOINT); // ❌ Error: Cannot find name 'AUTH_ENDPOINT'.
