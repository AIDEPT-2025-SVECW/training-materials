Frontend · Presentation Layer
React.js
What users see and interact with — runs in the browser
▾

What it does

  • Builds the user interface using reusable Components
  • Manages UI state with useState and useEffect hooks
  • Fetches data from the backend via REST API calls
  • Re-renders only changed parts using the Virtual DOM
  • Handles routing client-side with React Router

Key concepts

  • JSX — HTML-like syntax inside JavaScript
  • Props — data passed from parent to child component
  • State — component's own memory that triggers re-renders
  • Hooks — functions to plug into React's lifecycle
  • Context API — share state across many components
// React component — fetches and displays user data import { useState, useEffect } from 'react'; function UserList() { const [users, setUsers] = useState([]); useEffect(async () => { const res = await fetch('/api/users'); // → Express API const data = await res.json(); setUsers(data); }, []); return ( <ul> {users.map(u => <li key={u._id}>{u.name}</li>)} </ul> ); }

Backend · Application Layer
Node.js + Express
Business logic & API routes — runs on the server
▾

Node.js — the runtime

  • Runs JavaScript on the server (outside the browser)
  • Non-blocking, event-driven I/O — handles thousands of requests
  • Hosts the Express application and all server code
  • Built on Chrome's V8 JavaScript engine
  • npm ecosystem — 2 million+ packages available

Express — the API framework

  • Defines routes: GET /api/users, POST /api/users, etc.
  • Middleware pipeline — auth, validation, logging, CORS
  • Receives JSON requests from React, returns JSON responses
  • Calls Mongoose to read/write MongoDB data
  • Minimal and unopinionated — you decide the structure
// Express server — defines REST API routes const express = require('express'); const User = require('./models/User'); // Mongoose model const app = express(); app.use(express.json()); // parse JSON body // GET /api/users → fetch all users from MongoDB app.get('/api/users', async (req, res) => { const users = await User.find(); res.json(users); }); // POST /api/users → create a new user app.post('/api/users', async (req, res) => { const user = new User(req.body); await user.save(); // → MongoDB res.status(201).json(user); }); app.listen(5000);

Database · Persistence Layer
MongoDB
Stores data as flexible JSON-like documents — no fixed schema
▾

What makes it different

  • Documents instead of rows — each record is a JSON object
  • Collections instead of tables — no rigid schema required
  • Nested objects & arrays stored naturally — no JOIN queries
  • Scales horizontally — sharding across many servers
  • Atlas cloud — managed hosting with free tier available

Mongoose (the bridge)

  • ODM library — defines Schemas for your documents
  • Validates data before saving to the database
  • Provides clean JS methods: .find(), .save(), .deleteOne()
  • Supports middleware hooks (beforeSave, afterFind, etc.)
  • Turns raw MongoDB results into JavaScript objects
// Mongoose schema + model — defines the data shape const mongoose = require('mongoose'); const userSchema = new mongoose.Schema({ name: { type: String, required: true }, email: { type: String, unique: true }, createdAt: { type: Date, default: Date.now }, role: { type: String, enum: ['user', 'admin'] }, }); // A document in MongoDB looks like: // { _id: "64a1...", name: "Ravi", email: "ravi@..." } module.exports = mongoose.model('User', userSchema);

// Step-by-step data journey

1
React · Frontend
User action triggers a fetch()
A button click or page load in the React component calls fetch('/api/users') — an HTTP GET request leaves the browser.
// Inside a React useEffect or event handler const res = await fetch('/api/users'); const data = await res.json(); setUsers(data); // triggers re-render
2
Express · Backend
Express router matches the route
The HTTP request arrives at the Node.js server. Express checks its route table: GET /api/users matches. Middleware runs (auth check, logging), then the route handler is called.
// Express matches this route definition app.get('/api/users', authMiddleware, async (req, res) => { // handler runs here ↓ });
3
Node.js · Runtime
Handler queries the database via Mongoose
Inside the route handler, Node.js code calls User.find() using Mongoose. This sends a query to MongoDB and awaits the result — non-blocking, so other requests continue.
const users = await User.find() .select('name email') .sort({ createdAt: -1 }) .limit(50);
4
MongoDB · Database
MongoDB executes the query, returns documents
MongoDB searches the users collection, applies the filter, and returns matching documents as BSON (Binary JSON). Mongoose converts them to plain JavaScript objects.
// What MongoDB stores and returns [ { _id: "64a1...", name: "Ravi Kumar", email: "ravi@..." }, { _id: "64a2...", name: "Priya S", email: "priya@..." } ]
5
React · Frontend
React receives JSON and updates the UI
Express sends the data back as a JSON response. React's setUsers(data) updates component state, and React re-renders only the changed parts of the DOM — the user list appears instantly.
// React re-renders with new data return ( <ul> {users.map(u => <li key={u._id}>{u.name}</li>)} </ul> );

Why JavaScript at Every Layer?

The MERN advantage — one language, one mindset, front to back

What you get MERN (JS everywhere) Traditional Stack (PHP + SQL + JS)
Same language frontend & backend ✓ JavaScript ✗ PHP + JavaScript
Share code (types, utils, validation) ✓ Yes ✗ Duplicate in two languages
Same data format (JSON) end to end ✓ JSON → API → DB ≈ Partial (SQL rows ≠ JSON)
Full-stack developer productivity ✓ One developer, all layers ✗ Separate frontend / backend skills
Flexible data schema ✓ MongoDB (schema-less) ✗ SQL requires migration scripts
Real-time capability ✓ Node.js event loop + Socket.io ≈ Harder to implement
