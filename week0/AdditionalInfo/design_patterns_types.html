üéØ Types of Design Patterns

A comprehensive guide to the three main categories of software design patterns and their individual patterns

Design patterns are reusable solutions to commonly occurring problems in software design. They represent best practices evolved over time by experienced developers. The Gang of Four (GoF) identified 23 fundamental patterns, which are organized into three main categories based on their purpose and the problems they solve.

Memento
Captures and externalizes an object's internal state so it can be restored later without violating encapsulation.
Common Use Cases
Undo functionality, save states, snapshots, checkpoints
class Memento {
  constructor(state) {
    this.state = state;
  }
  getState() { return this.state; }
}

Observer
Defines a subscription mechanism to notify multiple objects about events that happen to the object they're observing.
Common Use Cases
Event systems, MVC architecture, reactive programming
class Subject {
  constructor() { this.observers = []; }
  notify(data) { this.observers.forEach(o => o.update(data)); }
}

üèõÔ∏è
State
Allows an object to alter its behavior when its internal state changes, appearing as if the object changed its class.
Common Use Cases
State machines, game characters, workflow systems
class State {
  handle(context) {
    // behavior based on current state
  }
}
Easy
üéØ
Strategy
Defines a family of algorithms, encapsulates each one, and makes them interchangeable at runtime.
Common Use Cases
Sorting algorithms, payment methods, compression algorithms
class Context {
  constructor(strategy) {
    this.strategy = strategy;
  }
  execute() { return this.strategy.algorithm(); }
}
Medium
üìã
Template Method
Defines the skeleton of an algorithm in a base class, letting subclasses override specific steps without changing structure.
Common Use Cases
Data processing pipelines, frameworks, code generation
class AbstractClass {
  templateMethod() {
    this.step1();
    this.step2(); // override in subclass
  }
}
Medium
üö∂
Visitor
Represents operations to be performed on elements of an object structure without changing the classes.
Common Use Cases
Compiler design, AST traversal, data export, reporting
class Visitor {
  visitConcreteElement(element) {
    // perform operation on element
  }
}

Easy
üîê
Singleton
Ensures a class has only one instance and provides a global point of access to it.
Common Use Cases
Database connections, logging, configuration settings, thread pools
class Singleton {
  constructor() {
    if (Singleton.instance) {
      return Singleton.instance;
    }
    Singleton.instance = this;
  }
}
Medium
üè≠
Factory Method
Creates objects without specifying the exact class to create. Defines an interface for creating objects.
Common Use Cases
UI element creation, database drivers, payment processors
class VehicleFactory {
  static create(type) {
    if (type === 'car') return new Car();
    if (type === 'bike') return new Bike();
  }
}
Hard
üèõÔ∏è
Abstract Factory
Provides an interface for creating families of related objects without specifying their concrete classes.
Common Use Cases
Cross-platform UI components, database abstraction layers
class UIFactory {
  createButton() { /* abstract */ }
  createTextbox() { /* abstract */ }
}
Medium
üë∑
Builder
Constructs complex objects step by step. Allows you to produce different types and representations.
Common Use Cases
SQL query builders, configuration objects, complex UI components
class QueryBuilder {
  select(fields) { return this; }
  from(table) { return this; }
  where(condition) { return this; }
}
Easy
üß¨
Prototype
Creates objects by cloning existing instances rather than creating new ones from scratch.
Common Use Cases
Game object spawning, document templates, configuration copying
class Prototype {
  clone() {
    return Object.create(this);
  }
}

Structural Patterns

Deal with object composition and relationships between entities

Easy
üîå
Adapter
Allows incompatible interfaces to work together. Acts as a bridge between two incompatible interfaces.
Common Use Cases
Third-party library integration, legacy system integration
class Adapter {
  constructor(adaptee) {
    this.adaptee = adaptee;
  }
  request() { return this.adaptee.specificRequest(); }
}
Medium
üåâ
Bridge
Separates an abstraction from its implementation, allowing both to vary independently.
Common Use Cases
Cross-platform development, device drivers, GUI frameworks
class Bridge {
  constructor(implementation) {
    this.implementation = implementation;
  }
}
Medium
üå≤
Composite
Composes objects into tree structures to represent part-whole hierarchies.
Common Use Cases
File systems, UI component trees, organizational structures
class Composite {
  constructor() { this.children = []; }
  add(child) { this.children.push(child); }
}
Easy
üé®
Decorator
Adds new functionality to objects dynamically without altering their existing structure.
Common Use Cases
Feature enhancement, middleware, plugin systems
class Decorator {
  constructor(component) {
    this.component = component;
  }
}
Easy
üé≠
Facade
Provides a simplified interface to a complex subsystem, hiding its complexity.
Common Use Cases
API wrappers, library interfaces, system integration

  operation() {
    this.subsystem1.operation();
    this.subsystem2.operation();
  }
}

Minimizes memory usage by sharing efficiently similar objects instead of storing all data in each object.
Common Use Cases
Game engines, text editors, graphics rendering
class FlyweightFactory {
  static flyweights = {};
  static getFlyweight(key) { /* ... */ }
}
Medium

Common Use Cases
Lazy loading, access control, caching, remote object access

  constructor(target) {
    this.target = target;
  }
  request() { /* control access */ }

Undo/redo operations, macro recording, queued operations

Collection traversal, data streaming, pagination

  next() { return this.items[this.index++]; }
