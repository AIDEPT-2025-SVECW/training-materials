1. Eager Initialization

The simplest form of Singleton pattern where the instance is created at class loading time. The instance is created whether it's needed or not.

This approach is thread-safe by default because the JVM handles the class loading and static variable initialization in a thread-safe manner.

Java Implementation

public class EagerSingleton {
    // Instance created at class loading time
    private static final EagerSingleton INSTANCE = new EagerSingleton();
    
    // Private constructor prevents instantiation
    private EagerSingleton() {
        // Prevent reflection attacks
        if (INSTANCE != null) {
            throw new RuntimeException("Use getInstance() method");
        }
    }
    
    // Global access point
    public static EagerSingleton getInstance() {
        return INSTANCE;
    }
    
    public void doSomething() {
        System.out.println("Eager Singleton is working!");
    }
}

‚úÖ Advantages

  • Simple and straightforward implementation
  • Thread-safe by default
  • No synchronization overhead at runtime
  • JVM guarantees thread safety during class loading

‚ùå Disadvantages

  • Instance created even if never used
  • May increase application startup time
  • Cannot handle exceptions during construction
  • Wastes memory if instance is not needed

2. Lazy Initialization

The instance is created only when it's first requested. This saves memory and improves startup time, but introduces thread safety concerns.

This basic lazy approach is NOT thread-safe and can create multiple instances in multi-threaded environments.

Java Implementation

public class LazySingleton {
    // Instance created only when needed
    private static LazySingleton instance;
    
    private LazySingleton() {
        // Simulate some initialization work
        try {
            Thread.sleep(100);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
    
    // NOT thread-safe!
    public static LazySingleton getInstance() {
        if (instance == null) {
            instance = new LazySingleton();
        }
        return instance;
    }
    
    public void doSomething() {
        System.out.println("Lazy Singleton is working!");
    }
}

‚úÖ Advantages

  • Instance created only when needed
  • Faster application startup
  • Memory efficient
  • Can handle exceptions during construction

‚ùå Disadvantages

  • NOT thread-safe
  • Multiple instances possible in multi-threaded environment
  • Race condition during instance creation
  • Unpredictable behavior in concurrent scenarios

3. Thread-Safe Singleton

We synchronize the getInstance() method to ensure thread safety. Only one thread can access the method at a time.

This approach is thread-safe but has performance implications due to synchronization overhead on every method call.

Java Implementation

public class ThreadSafeSingleton {
    private static ThreadSafeSingleton instance;
    
    private ThreadSafeSingleton() {
        // Simulate some initialization work
        try {
            Thread.sleep(100);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
    
    // Synchronized method - thread safe but slow
    public static synchronized ThreadSafeSingleton getInstance() {
        if (instance == null) {
            instance = new ThreadSafeSingleton();
        }
        return instance;
    }
    
    public void doSomething() {
        System.out.println("Thread-safe Singleton is working!");
    }
}

‚úÖ Advantages

  • Completely thread-safe
  • Lazy initialization preserved
  • Guarantees single instance
  • Simple synchronization approach

‚ùå Disadvantages

  • Performance overhead on every call
  • Synchronization needed even after initialization
  • Can become bottleneck in high-concurrency scenarios
  • Unnecessarily slow for subsequent calls

4. Double-Check Locking

Optimizes the thread-safe approach by only synchronizing during instance creation. Uses double-checking to minimize synchronization overhead.

The volatile keyword ensures that changes to the instance variable are visible to all threads immediately.

Java Implementation

public class DoubleCheckSingleton {
    // volatile ensures changes are visible to all threads
    private static volatile DoubleCheckSingleton instance;
    
    private DoubleCheckSingleton() {
        // Prevent reflection attacks
        if (instance != null) {
            throw new RuntimeException("Use getInstance() method");
        }
    }
    
    public static DoubleCheckSingleton getInstance() {
        // First check without synchronization
        if (instance == null) {
            // Synchronize only when needed
            synchronized (DoubleCheckSingleton.class) {
                // Double-check inside synchronized block
                if (instance == null) {
                    instance = new DoubleCheckSingleton();
                }
            }
        }
        return instance;
    }
    
    public void doSomething() {
        System.out.println("Double-check Singleton is working!");
    }
}

‚úÖ Advantages

  • Thread-safe and high performance
  • Lazy initialization maintained
  • Minimal synchronization overhead
  • Scales well in high-concurrency scenarios

‚ùå Disadvantages

  • Complex implementation
  • Requires volatile keyword (Java 5+)
  • Can be tricky to implement correctly
  • Still vulnerable to reflection attacks

5. Serialization-Safe Singleton

Handles serialization and deserialization correctly to maintain singleton property. Without proper handling, deserialization creates new instances.

The readResolve() method ensures that deserialization returns the same instance, and we also protect against cloning.

Java Implementation

import java.io.Serializable;
import java.io.ObjectStreamException;

public class SerializationSafeSingleton implements Serializable, Cloneable {
    private static final long serialVersionUID = 1L;
    
    private static volatile SerializationSafeSingleton instance;
    
    private SerializationSafeSingleton() {
        // Prevent reflection attacks
        if (instance != null) {
            throw new RuntimeException("Use getInstance() method");
        }
    }
    
    public static SerializationSafeSingleton getInstance() {
        if (instance == null) {
            synchronized (SerializationSafeSingleton.class) {
                if (instance == null) {
                    instance = new SerializationSafeSingleton();
                }
            }
        }
        return instance;
    }
    
    // Prevent creating new instance during deserialization
    protected Object readResolve() throws ObjectStreamException {
        return getInstance();
    }
    
    // Prevent cloning
    @Override
    protected Object clone() throws CloneNotSupportedException {
        throw new CloneNotSupportedException("Cloning is not allowed");
    }
    
    public void doSomething() {
        System.out.println("Serialization-safe Singleton is working!");
    }
}
