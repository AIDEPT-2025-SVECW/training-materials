Explore Java Web Application Configuration Concepts

SC
Servlet Config
Configuration parameters specific to individual servlets. Each servlet can have its own init parameters. 
<servlet>
    <servlet-name>MyServlet</servlet-name>
    <servlet-class>com.example.MyServlet</servlet-class>
    <init-param>
        <param-name>database-url</param-name>
        <param-value>jdbc:mysql://localhost:3306/mydb</param-value>
    </init-param>
    <init-param>
        <param-name>max-connections</param-name>
        <param-value>10</param-value>
    </init-param>
</servlet>

Usage in Servlet:

String dbUrl = getServletConfig().getInitParameter("database-url");
String maxConn = getServletConfig().getInitParameter("max-connections");
SX
Servlet Context
Global configuration parameters shared across all servlets in the web application. 
<context-param>
    <param-name>application-name</param-name>
    <param-value>My Web Application</param-value>
</context-param>
<context-param>
    <param-name>admin-email</param-name>
    <param-value>admin@example.com</param-value>
</context-param>
                    

                    
Usage in Servlet:

                    
String appName = getServletContext().getInitParameter("application-name");
String adminEmail = getServletContext().getInitParameter("admin-email");
S
Servlet
Java classes that handle HTTP requests and responses. Mapped to specific URL patterns. 
<servlet>
    <servlet-name>HomeServlet</servlet-name>
    <servlet-class>com.example.HomeServlet</servlet-class>
    <load-on-startup>1</load-on-startup>
</servlet>

<servlet-mapping>
    <servlet-name>HomeServlet</servlet-name>
    <url-pattern>/home</url-pattern>
</servlet-mapping>

<servlet-mapping>
    <servlet-name>HomeServlet</servlet-name>
    <url-pattern>*.do</url-pattern>
</servlet-mapping>

Key Points:

  • load-on-startup: Servlet loading priority
  • url-pattern: URL mapping patterns
  • Multiple mappings allowed per servlet
F
Filters
Interceptors that can modify requests and responses before they reach servlets. 
<filter>
    <filter-name>AuthenticationFilter</filter-name>
    <filter-class>com.example.AuthenticationFilter</filter-class>
    <init-param>
        <param-name>exclude-urls</param-name>
        <param-value>/login,/register</param-value>
    </init-param>
</filter>

<filter-mapping>
    <filter-name>AuthenticationFilter</filter-name>
    <url-pattern>/*</url-pattern>
</filter-mapping>

Common Filter Types:

  • Authentication & Authorization
  • Logging & Auditing
  • Character Encoding
  • CORS Headers
FC
Filter Chaining
Multiple filters can be applied to the same URL pattern, creating a filter chain. 
<!-- Filter 1: Logging -->
<filter>
    <filter-name>LoggingFilter</filter-name>
    <filter-class>com.example.LoggingFilter</filter-class>
</filter>

<!-- Filter 2: Authentication -->
<filter>
    <filter-name>AuthFilter</filter-name>
    <filter-class>com.example.AuthFilter</filter-class>
</filter>

<!-- Filter 3: Encoding -->
<filter>
    <filter-name>EncodingFilter</filter-name>
    <filter-class>com.example.EncodingFilter</filter-class>
</filter>

<!-- Mappings (Order matters!) -->
<filter-mapping>
    <filter-name>LoggingFilter</filter-name>
    <url-pattern>/*</url-pattern>
</filter-mapping>

<filter-mapping>
    <filter-name>AuthFilter</filter-name>
    <url-pattern>/secure/*</url-pattern>
</filter-mapping>

<filter-mapping>
    <filter-name>EncodingFilter</filter-name>
    <url-pattern>/*</url-pattern>
</filter-mapping>
W
Welcome Files
Default files served when a directory is requested without specifying a filename. 
<welcome-file-list>
    <welcome-file>index.html</welcome-file>
    <welcome-file>index.jsp</welcome-file>
    <welcome-file>home.jsp</welcome-file>
    <welcome-file>default.html</welcome-file>
</welcome-file-list>

How it works:

  • http://example.com/myapp/ ‚Üí looks for index.html
  • If not found ‚Üí looks for index.jsp
  • If not found ‚Üí looks for home.jsp
  • If not found ‚Üí looks for default.html
  • If none found ‚Üí 404 error or directory listing
L
Login Config
Configures authentication mechanism for the web application. 
<!-- Form-based Authentication -->
<login-config>
    <auth-method>FORM</auth-method>
    <form-login-config>
        <form-login-page>/login.jsp</form-login-page>
        <form-error-page>/login-error.jsp</form-error-page>
    </form-login-config>
</login-config>

<!-- Basic Authentication -->
<login-config>
    <auth-method>BASIC</auth-method>
    <realm-name>My Application</realm-name>
</login-config>

<!-- Security Constraints -->
<security-constraint>
    <web-resource-collection>
        <web-resource-name>Protected Area</web-resource-name>
        <url-pattern>/admin/*</url-pattern>
        <http-method>GET</http-method>
        <http-method>POST</http-method>
    </web-resource-collection>
    <auth-constraint>
        <role-name>admin</role-name>
        <role-name>manager</role-name>
    </auth-constraint>
</security-constraint>
SS
Session Config
Configures HTTP session behavior including timeout, tracking modes, and cookie settings. 
<session-config>
    <session-timeout>30</session-timeout> <!-- minutes -->
    <cookie-config>
        <name>MYSESSIONID</name>
        <domain>example.com</domain>
        <path>/myapp</path>
        <comment>Session tracking cookie</comment>
        <http-only>true</http-only>
        <secure>true</secure>
        <max-age>1800</max-age> <!-- seconds -->
    </cookie-config>
    <tracking-mode>COOKIE</tracking-mode>
    <tracking-mode>URL</tracking-mode>
</session-config>

Key Parameters:

  • session-timeout: Session expiry time in minutes
  • http-only: Prevents JavaScript access to cookies
  • secure: Cookies only sent over HTTPS
  • tracking-mode: COOKIE, URL, or SSL
