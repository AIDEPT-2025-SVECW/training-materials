01 · Foundation

Why Do We Need Exception Handling?

In real-world programs, things go wrong — a file is missing, a network is down, a user types a word where a number was expected. Without exception handling, Python prints an error and crashes immediately.

🔌

Runtime Errors

Errors that occur while the program runs — cannot be caught at compile time.

🛡️

Graceful Degradation

Instead of crashing, show a useful message or try a fallback.

📋

Controlled Flow

Decide what to do when something breaks — log, retry, or raise again.

🔒

Resource Safety

Ensure files/DB connections close even if an error occurs mid-operation. 

# Without exception handling — program CRASHES
numerator = 100
denominator = 0
result = numerator / denominator   # ZeroDivisionError!
# Everything below this line never runs
print("Result:", result)

# With exception handling — program CONTINUES
try:
    result = numerator / denominator
except ZeroDivisionError:
    print("⚠️  Cannot divide by zero! Using default value.")
    result = 0

print("Result:", result)  # This line executes safely
02 · Hierarchy

The Exception Family Tree

All exceptions in Python inherit from BaseException. Understanding the hierarchy helps you catch the right level.

🌳 BaseException
├── SystemExit ← raised by sys.exit()
├── KeyboardInterrupt ← Ctrl+C
└── Exception ← most errors you'll handle ▾
├── ArithmeticError
├── ZeroDivisionError   ← 10 / 0
└── OverflowError
├── LookupError
├── IndexError   ← list[99] on 3-item list
└── KeyError     ← dict['missing_key']
├── TypeError   ← wrong type operation
├── ValueError  ← right type, wrong value
├── AttributeError ← obj has no attribute
├── ImportError
├── FileNotFoundError
├── NameError
└── … and your custom exceptions!
⚠️

Catch specific before general! If you catch Exception before ZeroDivisionError, the specific handler never runs. Python checks except clauses top to bottom.

03 · Core Syntax

try / except / else / finally

The full structure has four optional parts. Each serves a distinct purpose:

Clause When it runs Typical use
try Always (first) The risky code
except Only if an exception occurs Handle the error
else Only if NO exception occurred Success path (keep try clean)
finally Always (last), no matter what Cleanup — close files, DB, network 
# Real-world example: Reading a student marks file
def get_student_marks(filename, student_name):
    try:
        f = open(filename, 'r')
        data = eval(f.read())          # dict of name → marks
        marks = data[student_name]      # could be KeyError
        percentage = marks / 500 * 100  # could be TypeError

    except FileNotFoundError:
        print(f"❌ File '{filename}' not found.")
        return None

    except KeyError:
        print(f"❌ Student '{student_name}' not in records.")
        return None

    except (TypeError, ValueError) as e:
        print(f"❌ Data error: {e}")
        return None

    else:
        # Only runs if no exception occurred
        print(f"✅ {student_name}: {marks}/500 ({percentage:.1f}%)")
        return percentage

    finally:
        # Always runs — clean up resources
        try: f.close()
        except: pass
        print("📂 File operation complete.")

04 · Advanced Patterns

raise, re-raise & Exception Chaining

💡 Analogy — Exception Chaining

Imagine a post office (your function) that receives a wrong-address letter. The postman doesn't just say "delivery failed" — he staples the original wrong address to the return slip so the sender knows exactly what went wrong and where. raise X from Y works the same way: it chains the cause to the new exception so the full story is preserved. 

# 1. raise — trigger an exception manually
def withdraw(balance, amount):
    if amount > balance:
        raise ValueError(f"Cannot withdraw ₹{amount}. Balance: ₹{balance}")
    return balance - amount

# 2. re-raise — catch, log, then bubble up
def process_payment(balance, amount):
    try:
        withdraw(balance, amount)
    except ValueError as e:
        print("[LOG] Payment failed:", e)
        raise   # re-raise the same exception

# 3. Exception chaining — from keyword
def load_config(path):
    try:
        with open(path) as f:
            return eval(f.read())
    except FileNotFoundError as e:
        raise RuntimeError("App cannot start without config") from e
        # Output shows: "The above exception was the direct cause of..."

Multiple except Blocks & Catching Groups 

# Catching multiple exception types in one block
try:
    data = int(input("Enter marks: "))
except (ValueError, TypeError):
    print("Please enter a valid integer.")

# Inspect the exception object
try:
    x = 1 / 0
except ZeroDivisionError as e:
    print(f"Type  : {type(e).__name__}")   # ZeroDivisionError
    print(f"Args  : {e.args}")             # ('division by zero',)
    print(f"Msg   : {e}")                  # division by zero

# Bare except — catches EVERYTHING including KeyboardInterrupt (avoid!)
try:
    risky()
except:            # ⚠ avoid bare except in production
    pass
✅

Best Practice: Always catch the most specific exception first. Never use a bare except: in production code — it swallows KeyboardInterrupt and SystemExit, making your program impossible to stop cleanly.

05 · Custom Exceptions

Building Your Own Exception Classes

When built-in exceptions aren't descriptive enough for your domain (banking, e-commerce, hospital systems…), create your own. 

# ── Base custom exception for your application ──
class BankingError(Exception):
    """Base class for all banking-related errors."""
    pass

class InsufficientFundsError(BankingError):
    def __init__(self, balance, amount):
        self.balance = balance
        self.amount  = amount
        self.shortfall = amount - balance
        super().__init__(
            f"Need ₹{amount}, have ₹{balance}. Short by ₹{self.shortfall}."
        )

class AccountFrozenError(BankingError):
    def __init__(self, account_id):
        super().__init__(f"Account {account_id} is frozen. Contact branch.")

# ── Using them ──
def transfer(from_acc, amount):
    if from_acc.frozen:
        raise AccountFrozenError(from_acc.id)
    if from_acc.balance < amount:
        raise InsufficientFundsError(from_acc.balance, amount)
    from_acc.balance -= amount

try:
    transfer(my_account, 50000)
except InsufficientFundsError as e:
    print(f"❌ {e}")
    print(f"   You need ₹{e.shortfall} more.")
except AccountFrozenError as e:
    print(f"🔒 {e}")
except BankingError as e:
    # catches any banking error not caught above
    print(f"🏦 {e}")
🎯

Descriptive

Name tells you exactly what failed — no guessing from generic messages.

📦

Extra Attributes

Attach domain data (balance, shortfall) to the exception object itself.

🌳

Hierarchy-aware

Catch BankingError to handle all banking errors in one place.

06 · Context Managers

The with Statement — Exception-safe Resources

💡 Analogy — The Dhaba Thali

When you sit down at a dhaba, the waiter sets your thali (opens a resource). When you're done eating — whether you finished everything or had to leave early (exception) — the waiter clears the table (closes the resource). You don't have to remind him. The with statement works exactly like this: guaranteed setup and guaranteed cleanup, no matter what happens in between. 

# ── Without context manager — risky! ──
f = open('students.txt', 'w')
try:
    f.write("Ravi, 92\nPreeti, 88\nArjun, 76\n")
finally:
    f.close()   # must remember this!

# ── With context manager — clean & safe ──
with open('students.txt', 'w') as f:
    f.write("Ravi, 92\nPreeti, 88\nArjun, 76\n")
# File is AUTOMATICALLY closed here — even if an exception occurred

# ── Build your own context manager ──
class DatabaseConnection:
    def __enter__(self):
        print("🔌 Connecting to database...")
        self.conn = create_connection()
        return self.conn   # this is the 'as' variable

    def __exit__(self, exc_type, exc_val, exc_tb):
        self.conn.close()
        print("🔒 Connection closed.")
        return False   # False = don't suppress exceptions

with DatabaseConnection() as db:
    db.query("SELECT * FROM students")
# __exit__ called automatically ↑

# ── Using contextlib — simpler approach ──
from contextlib import contextmanager

@contextmanager
def managed_connection():
    print("Connecting...")
    conn = create_connection()
    try:
        yield conn        # everything before yield = __enter__
    finally:
        conn.close()      # everything in finally = __exit__
        print("Disconnected.")
Approach Pros Best for
try/finally Explicit, no imports One-off cases
with + class Reusable, full control DB connections, locks
@contextmanager Concise, generator-based Simpler resources
