REQUIRED (Default)
Restaurant Analogy: You walk into a restaurant. If friends are already at a table, you join them. If not, you get a new table for your group.
Behavior: Joins existing transaction or creates new one if none exists.
Use Case: Most common scenario - standard business operations.
üè†
REQUIRES_NEW
Private Dining Room: You always book a private dining room, even if friends are dining in the main restaurant. Your meal is completely separate.
Behavior: Always creates a new transaction, suspending any existing one.
Use Case: Logging, audit trails, or operations that must complete regardless of main transaction.
ü§ù
SUPPORTS
Flexible Friend: You're happy to join friends at their table if they're already dining, but you're also fine eating alone without reserving a table.
Behavior: Joins existing transaction if present, runs without transaction if none exists.
Use Case: Read-only operations that can benefit from transactions but don't require them.
üö´
NOT_SUPPORTED
Antisocial Diner: You prefer to eat alone at the counter. If friends invite you to their table, you politely decline and eat separately.
Behavior: Always runs without transaction, suspending any existing one.
Use Case: Operations that shouldn't be part of transactions (like calling external services).

NEVER
Allergic to Restaurants: You have a severe allergy to restaurants. If someone tries to drag you to one, you get violently ill (throw an exception).
Behavior: Throws exception if transaction exists, runs normally if no transaction.
Use Case: Operations that must never run in a transaction context.
üéØ
MANDATORY
VIP Club: You can only enter if you're already a member (have a transaction). No transaction? You're not getting in!
Behavior: Requires existing transaction, throws exception if none exists.
Use Case: Operations that must always run within an existing transaction.
üìÑ
NESTED
Russian Dolls: You create a smaller transaction inside a bigger one, like a doll within a doll. Each can be committed/rolled back independently.
Behavior: Creates nested transaction (savepoint) if transaction exists, or new transaction if none.
Use Case: Partial rollback scenarios (not widely supported).
