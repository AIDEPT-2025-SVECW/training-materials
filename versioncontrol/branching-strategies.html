üè∑Ô∏è What Are Tags?

Tags are like bookmarks in your Git history. They mark specific points in time, usually for releases.

Timeline with Tags:
A ‚Üí B ‚Üí C ‚Üí D ‚Üí E ‚Üí F
        v1.0        v2.0
git tag v1.0.0 # Create a lightweight tag
git tag -a v1.0.0 -m "Release version 1.0.0" # Annotated tag
git push origin v1.0.0 # Push tag to remote

üöÄ Popular Branching Strategies

üåä Git Flow

The classic branching model with master, develop, feature, release, and hotfix branches

How it works:

  • master: Production-ready code
  • develop: Integration branch for features
  • feature/*: New features
  • release/*: Preparing releases
  • hotfix/*: Emergency fixes
git flow init
git flow feature start new-feature
git flow feature finish new-feature

üêô GitHub Flow

Simple workflow: master branch + feature branches with pull requests

Simple and effective:

  • master: Always deployable
  • feature branches: For all changes
  • Pull requests: For code review
  • Deploy: Immediately after merging
git checkout -b feature-branch
# Make changes, commit
git push origin feature-branch
# Create pull request

ü¶ä GitLab Flow

Environment-based branching with production and pre-production branches

Environment-focused:

  • master: Main development branch
  • production: What's live in production
  • pre-production: Staging environment
  • feature branches: Merge to master
git checkout -b feature/new-ui
# Develop ‚Üí merge to master ‚Üí deploy to pre-prod ‚Üí deploy to prod

1Ô∏è‚É£ OneFlow

Simplified Git Flow with only master branch and feature branches

Keep it simple:

  • master: Main branch for everything
  • feature branches: For development
  • Tags: Mark releases
  • Rebase: Keep history clean
git checkout -b feature/improvement
git rebase master
git checkout master && git merge feature/improvement
