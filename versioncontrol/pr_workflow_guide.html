🌟 What is a Pull Request?

A Pull Request (PR) is a method of submitting contributions to a project. It tells others about changes you've pushed to a branch in a repository and allows team members to review the changes before they're merged into the main codebase.

📋 Typical PR Lifecycle

  1. Branch Creation: Create a feature branch from main/master
  2. Development: Make changes and commit them
  3. Push: Push your branch to the remote repository
  4. PR Creation: Open a pull request
  5. Review Process: Code review and discussions
  6. Conflict Resolution: Handle any merge conflicts
  7. Approval: Get approval from reviewers
  8. Merge: Merge the PR into the target branch

📝 Creating a Pull Request

1

Create and Switch to Feature Branch

# Create and switch to a new branch git checkout -b feature/user-authentication # Or create branch first, then switch git branch feature/user-authentication git checkout feature/user-authentication # Using newer git switch command git switch -c feature/user-authentication
2

Make Changes and Commit

# Make your changes to files # Then stage the changes git add src/auth/login.js git add src/auth/register.js # Or stage all changes git add . # Commit with a descriptive message git commit -m "feat: implement user authentication system - Add login functionality with JWT tokens - Add user registration with email validation - Add password hashing with bcrypt - Add authentication middleware Closes #123"
💡 Commit Message Best Practices:
  • Use conventional commit format: type(scope): description
  • Types: feat, fix, docs, style, refactor, test, chore
  • Keep subject line under 50 characters
  • Reference issues/tickets when applicable
3

Push Branch to Remote

# Push branch to remote repository git push origin feature/user-authentication # If it's the first push of this branch, set upstream git push -u origin feature/user-authentication # Check remote branches git branch -r
4

Create Pull Request

GitHub PR Creation

  1. Navigate to your repository on GitHub
  2. Click "Compare & pull request" (appears after pushing)
  3. Or go to "Pull requests" tab → "New pull request"
  4. Select base branch (usually main/master) and compare branch
  5. Add title and description
  6. Add reviewers, assignees, and labels
  7. Click "Create pull request"

GitLab Merge Request Creation

  1. Navigate to your project on GitLab
  2. Go to "Merge requests" → "New merge request"
  3. Select source and target branches
  4. Add title and description
  5. Assign reviewers and set labels
  6. Configure merge options
  7. Click "Create merge request"

Bitbucket Pull Request Creation

  1. Navigate to your repository on Bitbucket
  2. Click "Create pull request"
  3. Select source and destination branches
  4. Add title and description
  5. Add reviewers
  6. Set up build status checks if needed
  7. Click "Create pull request"

1

Understanding Merge Conflicts

Merge conflicts occur when Git cannot automatically resolve differences between branches. This happens when:

  • The same line(s) are modified differently in both branches
  • A file is deleted in one branch but modified in another
  • A file is renamed in one branch but modified in another
2

Detecting Conflicts

# Update your branch with latest changes from main git checkout main git pull origin main git checkout feature/user-authentication # Try to merge or rebase git merge main # OR git rebase main # If conflicts occur, Git will show: # CONFLICT (content): Merge conflict in filename.js # Automatic merge failed; fix conflicts and then commit the result.
3

Conflict Markers Explained

<<<<<<< HEAD (Current Change) function authenticate(user) { return jwt.sign(user, 'secret-key'); } ======= function authenticateUser(user) { return jwt.sign(user, process.env.JWT_SECRET); } >>>>>>> feature/user-authentication (Incoming Change)
  • <<<<<<< HEAD: Your current branch changes
  • =======: Separator between changes
  • >>>>>>> branch-name: Incoming changes from other branch
4

Resolving Conflicts Manually

# 1. Open conflicted files in your editor # 2. Look for conflict markers (<<<<<<, ======, >>>>>>) # 3. Decide which changes to keep: # - Keep HEAD changes # - Keep incoming changes # - Keep both (merge them) # - Write completely new code # Example resolution: function authenticateUser(user) { return jwt.sign(user, process.env.JWT_SECRET); } # 4. Remove conflict markers # 5. Stage the resolved files git add src/auth/login.js # 6. Continue merge/rebase git commit # for merge git rebase --continue # for rebase
5

Using Merge Tools

# Configure merge tool (one-time setup) git config --global merge.tool vscode # OR git config --global merge.tool vimdiff # Launch merge tool for conflicts git mergetool # Popular merge tools: # - VS Code: code --wait # - Sublime Merge # - KDiff3 # - Meld # - P4Merge
6

Advanced Conflict Resolution

# Show conflict status git status # Show conflicts in more detail git diff --name-only --diff-filter=U # Abort merge if needed git merge --abort # Abort rebase if needed git rebase --abort # Use theirs or ours strategy for specific files git checkout --theirs filename.js # Use incoming changes git checkout --ours filename.js # Use your changes
⚠️ Conflict Resolution Tips:
  • Always test your code after resolving conflicts
  • Don't just pick one side - understand what each change does
  • Communicate with team members about complex conflicts
  • Consider pair programming for difficult conflict resolution

🔀 Merging Strategies

Merge Strategy Visualizer

1. 🔀 Merge Commit Strategy

Creates a merge commit that combines both branches

# Standard merge (creates merge commit) git checkout main git merge feature/user-auth # GitHub CLI merge gh pr merge 123 --merge
✅ When to use: When you want to preserve branch history and see exactly when features were integrated.
⚠️ Considerations: Can create a cluttered commit history with many merge commits.

2. 📦 Squash and Merge Strategy

Combines all commits from the feature branch into a single commit

# Interactive rebase to squash commits git checkout feature/user-auth git rebase -i main # In the interactive editor, change 'pick' to 'squash' for commits to combine # Then merge git checkout main git merge feature/user-auth # GitHub CLI squash merge gh pr merge 123 --squash
✅ When to use: For clean commit history, especially with messy feature branch commits.
⚠️ Considerations: Loses individual commit history from the feature branch.

3. 🔄 Rebase and Merge Strategy

Replays commits from feature branch onto main without creating a merge commit

# Rebase feature branch onto main git checkout feature/user-auth git rebase main # Fast-forward merge (no merge commit) git checkout main git merge feature/user-auth # GitHub CLI rebase merge gh pr merge 123 --rebase
✅ When to use: For linear commit history without merge commits.
⚠️ Considerations: Rewrites commit history, can be dangerous on shared branches.

🎯 Choosing the Right Strategy

Strategy History Best For
Merge Commit Preserves all commits + merge commit Feature branches, collaboration
Squash Single commit per feature Clean history, small features
Rebase Linear history, no merge commits Simple changes, hotfixes

🚀 Advanced Merge Techniques

# Merge with custom commit message git merge feature/user-auth -m "Merge user authentication feature - Implements JWT-based authentication - Adds password hashing - Includes user registration flow" # Merge without fast-forward (always create merge commit) git merge --no-ff feature/user-auth # Dry run merge (see what would happen) git merge --no-commit --no-ff feature/user-auth git diff --cached # Review changes git merge --abort # Cancel if needed

🔍 Code Review Guidelines

👥 Reviewer Guidelines

  • Be constructive: Focus on the code, not the person
  • Explain your reasoning: Don't just say "this is wrong"
  • Suggest improvements: Offer alternative solutions
  • Ask questions: "Could we handle this edge case?"
  • Praise good code: Acknowledge clever solutions
  • Check for: Logic errors, security issues, performance problems
  • Test the changes: Pull the branch and test locally

✍️ Author Guidelines

  • Keep PRs small: Easier to review and less likely to have issues
  • Write clear commit messages: Follow conventional commit format
  • Add tests: Include unit tests for new functionality
  • Update documentation: Keep README and docs current
  • Respond promptly: Address review comments quickly
  • Be open to feedback: Don't take criticism personally
  • Self-review first: Review your own code before requesting review

🏗️ Branch Management

# Good branch naming conventions feature/user-authentication bugfix/login-error-handling hotfix/security-patch docs/api-documentation refactor/payment-service # Clean up after merge git branch -d feature/user-authentication # Delete local branch git push origin --delete feature/user-authentication # Delete remote branch # Prune deleted remote branches git remote prune origin

🔧 Troubleshooting Common Issues

❌ Common Problems & Solutions

Problem: "Your branch is behind origin/main"

# Solution: Update your branch git checkout main git pull origin main git checkout your-feature-branch git merge main # OR git rebase main

Problem: "Cannot merge due to conflicts"

# Solution: Resolve conflicts manually git status # See conflicted files # Edit conflicted files to resolve git add conflicted-file.js git commit -m "resolve merge conflicts"

Problem: "PR shows too many commits"

# Solution: Squash commits git rebase -i HEAD~3 # Squash last 3 commits # In editor: change 'pick' to 'squash' for commits to combine git push --force-with-lease origin your-branch

Problem: "Accidentally committed to main"

# Solution: Move commits to new branch git branch new-feature-branch # Create branch with current commits git reset --hard HEAD~2 # Reset main back 2 commits git checkout new-feature-branch # Switch to new branch

🛟 Emergency Commands

# Abort current merge git merge --abort # Abort current rebase git rebase --abort # Undo last commit (keep changes) git reset --soft HEAD~1 # Undo last commit (discard changes) git reset --hard HEAD~1 # Force push (use with caution!) git push --force-with-lease origin branch-name # See what you've been doing git reflog # Recover lost commits git checkout git checkout -b recovered-branch
⚠️ Use force push carefully: Only force push to feature branches, never to main/master or shared branches!
