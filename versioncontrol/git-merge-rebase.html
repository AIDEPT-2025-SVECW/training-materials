Git Merge

Git merge combines two branches by creating a new merge commit that has two parent commits. It preserves the complete history and chronological order of commits.

Basic Merge Command

# Switch to the branch you want to merge INTO
git checkout main

# Merge the feature branch into main
git merge feature

Merge Types

Fast-Forward Merge: When there are no new commits on the target branch, Git simply moves the pointer forward. 
# Fast-forward merge (no merge commit created)
git merge feature

# Force creation of merge commit even if fast-forward is possible
git merge --no-ff feature

Handling Merge Conflicts

# Attempt merge
git merge feature

# If conflicts occur, Git will pause and show:
# Auto-merging file.txt
# CONFLICT (content): Merge conflict in file.txt

# View conflicted files
git status

# Edit conflicted files (look for conflict markers)
# <<<<<<< HEAD
# your changes
# =======
# their changes
# >>>>>>> feature

# After resolving conflicts
git add file.txt
git commit -m "Merge feature branch, resolved conflicts"

# Or abort the merge
git merge --abort

Advanced Merge Options

# Merge with a custom commit message
git merge feature -m "Merge feature: Add new functionality"

# Merge without committing (review changes first)
git merge --no-commit feature

# Squash all feature branch commits into one
git merge --squash feature
git commit -m "Feature: Complete implementation"

# Use a specific merge strategy
git merge -s recursive -X theirs feature

When to Use Merge

  • Working on public/shared branches
  • Want to preserve complete history
  • Need to maintain context of when branches diverged
  • Working in teams where history transparency is important

Basic Rebase Command

# Switch to the feature branch
git checkout feature

# Rebase onto main
git rebase main

# Alternative: specify both branches
git rebase main feature

Handling Rebase Conflicts

# Start rebase
git rebase main

# If conflicts occur, Git pauses at each conflict
# Resolve conflicts in files

# After resolving, stage the files
git add file.txt

# Continue the rebase
git rebase --continue

# Skip the current commit if needed
git rebase --skip

# Abort and return to original state
git rebase --abort

Interactive Rebase

Interactive rebase allows you to edit, reorder, squash, or drop commits:

# Start interactive rebase for last 3 commits
git rebase -i HEAD~3

# Or rebase interactively onto another branch
git rebase -i main

# In the editor, you'll see:
# pick abc1234 First commit
# pick def5678 Second commit
# pick ghi9012 Third commit

# Change 'pick' to:
# p, pick   = use commit as-is
# r, reword = use commit, but edit the message
# e, edit   = use commit, but stop to amend
# s, squash = combine with previous commit
# f, fixup  = like squash, but discard commit message
# d, drop   = remove commit

Common Interactive Rebase Operations

# Squash last 3 commits into one
git rebase -i HEAD~3
# Change last 2 'pick' to 'squash', save and edit commit message

# Edit a specific commit
git rebase -i HEAD~5
# Change 'pick' to 'edit' for desired commit
# Make your changes
git add .
git commit --amend
git rebase --continue

# Reorder commits
git rebase -i HEAD~4
# Simply reorder the lines in the editor

Rebase with Autosquash

# Create a fixup commit
git commit --fixup abc1234

# Automatically squash fixup commits
git rebase -i --autosquash main

Force Push After Rebase

Warning: After rebasing a pushed branch, you must force push (rewrites history). 
# Force push (be careful with shared branches!)
git push --force-with-lease

# Or the more aggressive version
git push --force

When to Use Rebase

  • Working on private/local branches before pushing
  • Want a clean, linear project history
  • Cleaning up local commits before sharing
  • Updating feature branch with latest main changes
Golden Rule: Never rebase commits that have been pushed to a shared/public branch!

Decision Guide

Use MERGE when:

  • Working with public/shared branches
  • You need to preserve the exact history
  • Multiple people are working on the same branch
  • You want to maintain context of parallel development
  • Integrating long-lived feature branches
  • Following "merge commit" workflow in your team

Use REBASE when:

  • Working on private, local branches
  • You want a clean, linear history
  • Updating your feature branch with latest main
  • Cleaning up commits before pushing
  • Squashing multiple small commits
  • You haven't pushed the commits yet

Example Workflow Comparison

Merge Workflow:

# On feature branch
git checkout feature
git commit -m "Add feature A"
git commit -m "Add feature B"

# Update with main changes
git checkout main
git pull
git checkout feature
git merge main

# When feature is complete
git checkout main
git merge feature
git push

Rebase Workflow:

# On feature branch
git checkout feature
git commit -m "Add feature A"
git commit -m "Add feature B"

# Update with main changes
git fetch origin
git rebase origin/main

# Clean up commits locally
git rebase -i HEAD~2

# When feature is complete
git checkout main
git merge feature  # This will be a fast-forward
git push

Quick Reference Commands

# MERGE COMMANDS
git merge <branch>                    # Merge branch into current branch
git merge --no-ff <branch>            # Force merge commit
git merge --squash <branch>           # Squash and merge
git merge --abort                     # Cancel merge

# REBASE COMMANDS
git rebase <branch>                   # Rebase current branch onto branch
git rebase -i <branch>                # Interactive rebase
git rebase --continue                 # Continue after resolving conflicts
git rebase --skip                     # Skip current commit
git rebase --abort                    # Cancel rebase

# CHECKING STATUS
git status                            # See current state
git log --graph --oneline             # View commit graph
git log --all --decorate --graph      # Detailed graph

# CONFLICT RESOLUTION
git diff                              # See changes
git add <file>                        # Stage resolved files
git checkout --theirs <file>         # Accept their changes
git checkout --ours <file>           # Keep your changes

Best Practices Summary

  • Never rebase public history - only rebase commits that haven't been pushed
  • Use merge for feature integration - especially on shared branches
  • Use rebase to clean up local work - before pushing to remote
  • Communicate with your team - establish conventions
  • Use interactive rebase sparingly - to organize commits logically
  • Always pull before push after rebase - check for remote changes
