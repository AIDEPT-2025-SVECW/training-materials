Custom Properties & @ConfigurationProperties

üéØ What are Custom Properties?

Custom properties allow you to define application-specific configuration values that can be easily managed across different environments. Instead of hardcoding values in your application, you can externalize them to configuration files.

üî• Problem: Hardcoded values make applications inflexible and difficult to deploy across different environments.
‚úÖ Solution: Use custom properties with @ConfigurationProperties to create type-safe, organized configuration.

üèóÔ∏è @ConfigurationProperties Deep Dive

The @ConfigurationProperties annotation binds external configuration properties to a Java object, providing type safety and validation.

‚ùå Without @ConfigurationProperties

@Component public class AppConfig { @Value("${app.name}") private String appName; @Value("${app.version}") private String appVersion; @Value("${app.database.url}") private String databaseUrl; @Value("${app.database.username}") private String databaseUsername; // Many @Value annotations... }

Issues:
‚Ä¢ Repetitive @Value annotations
‚Ä¢ No type safety
‚Ä¢ Hard to organize related properties
‚Ä¢ No validation support

‚úÖ With @ConfigurationProperties

@ConfigurationProperties(prefix = "app") @Component public class AppConfig { private String name; private String version; private Database database = new Database(); // Getters and setters public static class Database { private String url; private String username; private String password; // Getters and setters } }

Benefits:
‚Ä¢ Clean, organized structure
‚Ä¢ Type-safe binding
‚Ä¢ Nested object support
‚Ä¢ Built-in validation

üí° Best Practices

1
Use Meaningful Prefixes: Choose clear, descriptive prefixes like 'app', 'security', 'database' rather than generic ones.
2
Group Related Properties: Organize related configuration into nested objects for better structure.
3
Add Validation: Use JSR-303 annotations to validate configuration values at startup.
4
Provide Default Values: Initialize fields with sensible defaults to avoid null pointer exceptions.
5
Document Properties: Use Javadoc comments to document what each property does.

Spring Boot Profiles

‚ùì Why Spring Boot Introduced Profiles

üö®The Core Problem

Before Spring Boot profiles, developers faced significant challenges when deploying applications across different environments. Each environment required different configurations, leading to several critical issues:

üî• Configuration Nightmare: Managing separate configuration files for development, testing, staging, and production environments was complex and error-prone.

üéØReal-World Scenarios

üè† Development Environment

‚Ä¢ Local database (H2, MySQL on localhost)
‚Ä¢ Debug logging enabled
‚Ä¢ Hot reload features
‚Ä¢ Mock external services

üß™ Testing Environment

‚Ä¢ Test database (separate schema)
‚Ä¢ Test-specific configurations
‚Ä¢ Mock email services
‚Ä¢ Different port numbers

üöÄ Production Environment

‚Ä¢ Production database (AWS RDS)
‚Ä¢ Error-level logging only
‚Ä¢ Real email SMTP servers
‚Ä¢ Security configurations

‚ö†Ô∏èProblems Without Profiles

1
Manual Configuration Switching: Developers had to manually change configuration files before deploying to different environments, leading to human errors.
2
Deployment Risks: Wrong configuration in production could cause security vulnerabilities or application failures.
3
Environment-Specific Builds: Teams had to create separate builds for each environment, increasing complexity.

üîÑBefore vs After Profiles

‚ùå Before Profiles

# dev-config.properties server.port=8080 database.url=jdbc:h2:mem:testdb logging.level.root=DEBUG # prod-config.properties server.port=80 database.url=jdbc:mysql://prod-db:3306/app logging.level.root=ERROR

Issues:
‚Ä¢ Manual file switching
‚Ä¢ Easy to deploy wrong config
‚Ä¢ No conditional bean loading

‚úÖ With Profiles

# application.yml server: port: 8080 --- spring: profiles: dev database: url: jdbc:h2:mem:testdb logging: level: root: DEBUG --- spring: profiles: prod database: url: jdbc:mysql://prod-db:3306/app logging: level: root: ERROR

Benefits:
‚Ä¢ Single file management
‚Ä¢ Environment-specific activation
‚Ä¢ Conditional bean creation

üéâSpring Boot Profiles Solution

üåü Profiles provide a clean, maintainable way to:
‚Ä¢ Segregate configuration for different environments
‚Ä¢ Conditionally load beans and components
‚Ä¢ Enable/disable features based on environment
‚Ä¢ Reduce deployment complexity and errors

Key Benefits:

  • Environment Isolation: Each environment gets its own configuration without affecting others
  • Conditional Bean Loading: Load specific beans only for certain environments
  • Feature Toggling: Enable/disable features based on active profiles
  • Simplified Deployment: One artifact can be deployed to multiple environments
  • Testing Support: Dedicated test profiles for different testing scenarios

üé≠ Profile Management

Profiles allow you to segregate parts of your application configuration and make it available only in certain environments.

üì® Mail Service Implementation

@Service
public class EmailService {
    
    @Autowired
    private JavaMailSender mailSender;
    
    public void sendSimpleEmail(String to, String subject, String text) {
        SimpleMailMessage message = new SimpleMailMessage();
        message.setTo(to);
        message.setSubject(subject);
        message.setText(text);
        mailSender.send(message);
    }
    
    public void sendHtmlEmail(String to, String subject, String htmlContent) throws MessagingException {
        MimeMessage message = mailSender.createMimeMessage();
        MimeMessageHelper helper = new MimeMessageHelper(message, true);
        
        helper.setTo(to);
        helper.setSubject(subject);
        helper.setText(htmlContent, true);
        
        mailSender.send(message);
    }
}

üîí Security & Best Practices

  • Use App Passwords: For Gmail, use app-specific passwords instead of your regular password
  • Environment Variables: Store credentials in environment variables, not in code
  • SSL/TLS: Always use encrypted connections (STARTTLS or SSL)
  • Rate Limiting: Implement rate limiting to prevent spam
  • Email Validation: Validate email addresses before sending
