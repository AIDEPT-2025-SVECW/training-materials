Custom Properties & @ConfigurationProperties

🎯 What are Custom Properties?

Custom properties allow you to define application-specific configuration values that can be easily managed across different environments. Instead of hardcoding values in your application, you can externalize them to configuration files.

🔥 Problem: Hardcoded values make applications inflexible and difficult to deploy across different environments.
✅ Solution: Use custom properties with @ConfigurationProperties to create type-safe, organized configuration.

🏗️ @ConfigurationProperties Deep Dive

The @ConfigurationProperties annotation binds external configuration properties to a Java object, providing type safety and validation.

❌ Without @ConfigurationProperties

@Component public class AppConfig { @Value("${app.name}") private String appName; @Value("${app.version}") private String appVersion; @Value("${app.database.url}") private String databaseUrl; @Value("${app.database.username}") private String databaseUsername; // Many @Value annotations... }

Issues:
• Repetitive @Value annotations
• No type safety
• Hard to organize related properties
• No validation support

✅ With @ConfigurationProperties

@ConfigurationProperties(prefix = "app") @Component public class AppConfig { private String name; private String version; private Database database = new Database(); // Getters and setters public static class Database { private String url; private String username; private String password; // Getters and setters } }

Benefits:
• Clean, organized structure
• Type-safe binding
• Nested object support
• Built-in validation

🎨 Advanced Configuration Features

✅ Validation Support

@ConfigurationProperties(prefix = "app")
@Validated
@Component
public class AppConfig {
    
    @NotBlank
    private String name;
    
    @Min(1)
    @Max(65535)
    private int port;
    
    @Email
    private String adminEmail;
    
    @Valid
    private Database database;
    
    // Getters and setters
}

🧪 Testing Configuration Properties

@SpringBootTest
@TestPropertySource(properties = {
    "app.name=Test Application",
    "app.version=1.0.0-TEST",
    "app.database.url=jdbc:h2:mem:testdb"
})
class AppConfigTest {
    
    @Autowired
    private AppConfig appConfig;
    
    @Test
    void testConfigurationProperties() {
        assertThat(appConfig.getName()).isEqualTo("Test Application");
        assertThat(appConfig.getVersion()).isEqualTo("1.0.0-TEST");
        assertThat(appConfig.getDatabase().getUrl()).isEqualTo("jdbc:h2:mem:testdb");
    }
}

💡 Best Practices

1
Use Meaningful Prefixes: Choose clear, descriptive prefixes like 'app', 'security', 'database' rather than generic ones.
2
Group Related Properties: Organize related configuration into nested objects for better structure.
3
Add Validation: Use JSR-303 annotations to validate configuration values at startup.
4
Provide Default Values: Initialize fields with sensible defaults to avoid null pointer exceptions.
5
Document Properties: Use Javadoc comments to document what each property does.

Spring Boot Profiles

❓ Why Spring Boot Introduced Profiles

🚨The Core Problem

Before Spring Boot profiles, developers faced significant challenges when deploying applications across different environments. Each environment required different configurations, leading to several critical issues:

🔥 Configuration Nightmare: Managing separate configuration files for development, testing, staging, and production environments was complex and error-prone.

🎯Real-World Scenarios

🏠 Development Environment

• Local database (H2, MySQL on localhost)
• Debug logging enabled
• Hot reload features
• Mock external services

🧪 Testing Environment

• Test database (separate schema)
• Test-specific configurations
• Mock email services
• Different port numbers

🚀 Production Environment

• Production database (AWS RDS)
• Error-level logging only
• Real email SMTP servers
• Security configurations

⚠️Problems Without Profiles

1
Manual Configuration Switching: Developers had to manually change configuration files before deploying to different environments, leading to human errors.
2
Deployment Risks: Wrong configuration in production could cause security vulnerabilities or application failures.
3
Environment-Specific Builds: Teams had to create separate builds for each environment, increasing complexity.

🔄Before vs After Profiles

❌ Before Profiles

# dev-config.properties server.port=8080 database.url=jdbc:h2:mem:testdb logging.level.root=DEBUG # prod-config.properties server.port=80 database.url=jdbc:mysql://prod-db:3306/app logging.level.root=ERROR

Issues:
• Manual file switching
• Easy to deploy wrong config
• No conditional bean loading

✅ With Profiles

# application.yml server: port: 8080 --- spring: profiles: dev database: url: jdbc:h2:mem:testdb logging: level: root: DEBUG --- spring: profiles: prod database: url: jdbc:mysql://prod-db:3306/app logging: level: root: ERROR

Benefits:
• Single file management
• Environment-specific activation
• Conditional bean creation

🎉Spring Boot Profiles Solution

🌟 Profiles provide a clean, maintainable way to:
• Segregate configuration for different environments
• Conditionally load beans and components
• Enable/disable features based on environment
• Reduce deployment complexity and errors

Key Benefits:

  • Environment Isolation: Each environment gets its own configuration without affecting others
  • Conditional Bean Loading: Load specific beans only for certain environments
  • Feature Toggling: Enable/disable features based on active profiles
  • Simplified Deployment: One artifact can be deployed to multiple environments
  • Testing Support: Dedicated test profiles for different testing scenarios

🎭 Profile Management

Profiles allow you to segregate parts of your application configuration and make it available only in certain environments.

⚙️ SpringApplication Configuration

You can customize the SpringApplication before running it:

@SpringBootApplication
public class MyApplication {
    
    public static void main(String[] args) {
        SpringApplication app = new SpringApplication(MyApplication.class);
        
        // Customize the application
        app.setBannerMode(Banner.Mode.OFF);
        app.setDefaultProperties(Collections.singletonMap("server.port", "8083"));
        app.setAdditionalProfiles("dev");
        
        app.run(args);
    }
}

🔧 Main Class Best Practices

📍 Package Structure

Place your main class in the root package to enable component scanning for all sub-packages.

com.example.myapp
├── MyApplication.java (main class)
├── controller/
├── service/
├── repository/
└── model/

🎯 Naming Convention

Follow consistent naming patterns for your main class.

  • Application suffix
  • App suffix
  • Descriptive project name

🚀 Startup Configuration

Keep the main method simple and move complex configuration to separate classes.

// Good practice
@SpringBootApplication
public class MyApp {
    public static void main(String[] args) {
        SpringApplication.run(MyApp.class, args);
    }
}

🔄 Application Events

Listen to application lifecycle events in your main class:

@SpringBootApplication
public class MyApplication implements ApplicationRunner {
    
    public static void main(String[] args) {
        SpringApplication.run(MyApplication.class, args);
    }
    
    @Override
    public void run(ApplicationArguments args) throws Exception {
        System.out.println("Application started successfully!");
    }
    
    @EventListener
    public void handleContextRefresh(ContextRefreshedEvent event) {
        System.out.println("Context refreshed: " + event.getTimestamp());
    }
}

📨 Mail Service Implementation

@Service
public class EmailService {
    
    @Autowired
    private JavaMailSender mailSender;
    
    public void sendSimpleEmail(String to, String subject, String text) {
        SimpleMailMessage message = new SimpleMailMessage();
        message.setTo(to);
        message.setSubject(subject);
        message.setText(text);
        mailSender.send(message);
    }
    
    public void sendHtmlEmail(String to, String subject, String htmlContent) throws MessagingException {
        MimeMessage message = mailSender.createMimeMessage();
        MimeMessageHelper helper = new MimeMessageHelper(message, true);
        
        helper.setTo(to);
        helper.setSubject(subject);
        helper.setText(htmlContent, true);
        
        mailSender.send(message);
    }
}

🔒 Security & Best Practices

  • Use App Passwords: For Gmail, use app-specific passwords instead of your regular password
  • Environment Variables: Store credentials in environment variables, not in code
  • SSL/TLS: Always use encrypted connections (STARTTLS or SSL)
  • Rate Limiting: Implement rate limiting to prevent spam
  • Email Validation: Validate email addresses before sending
